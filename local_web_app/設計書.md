# ピアノ教室運営アプリ Ver.3 設計書

## 構成

- `app_pages/`: 今日の受付、生徒・請求管理、例外処理、売上
- `services/calendar_service.py`: Calendar APIとモックの同一インターフェース
- `services/auth_service.py`: Supabase Google OAuth、セッション
- `services/v3_repository.py`: Supabase/SQLite切替と1ボタン処理
- `services/sales_service.py`: 確定申告向け集計
- 既存サービス: 生徒、請求、入金、監査、出力、バックアップ

## 主処理

`今日のGoogle予定取得 → 予定タイトルと生徒氏名の完全一致 → 未入金請求選択 → 受領・押印済み → payments追記 + charges更新 + audit_logs追記 → 売上集計`

クラウドの通常受付は `complete_lesson_payment` PostgreSQL関数で請求行をロックし、残額確認・入金・請求状態・監査を単一トランザクションで確定する。`calendar_event_id` の部分UNIQUE制約でも同一予定の再処理を拒否する。

通常受付は全額・押印済みだけを扱う。一部入金、後押印、取消、金額修正は例外画面へ隔離する。DB更新はサービス層に集約し、将来の出席、振替、通知、口座振替、PDF、発表会、コンクール機能を別サービスとして追加できる。

## クラウドと安全性

Supabase PostgreSQLを永続化先、Supabase Auth + Googleを本人確認に使用する。本番はHTTPSホストを前提とする。全業務テーブルでRLSを有効化し、匿名利用を拒否する。Google Calendar tokenはセッション中だけ保持する。公開リポジトリへDB、Secrets、バックアップを登録しない。

認可はSecretsの `allowed_emails` とSupabase `allowed_teachers` の二層で行う。氏名照合は空白と安全な接尾語だけを正規化し、一意でない候補を自動決定しない。手動対応は `calendar_mappings` に保存する。

## 可用性

外部設定なしではSQLite/モックモードで開発・テストできる。本番モードでCalendar取得に失敗した場合はエラーを明示し、架空予定へ自動フォールバックして誤登録することはない。

## 将来拡張

出席、振替、LINE、メール、口座振替、領収書、発表会、コンクールは独立テーブルとサービスを追加する。生徒・予定・請求のID境界を維持し、画面から外部APIを直接呼ばない。
