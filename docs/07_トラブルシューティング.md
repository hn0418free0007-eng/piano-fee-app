# トラブルシューティング

## 目的

実際に発生した問題と、安全な切り分け方法をまとめます。

最終更新日: 2026-07-17

## PKCE: code verifierが空

### 症状

```text
invalid request: both auth code and code verifier should be non-empty
```

URLには `?code=...` があるのにSupabaseセッション交換が失敗します。

### 原因

`supabase-py` が生成したPKCE verifierがメモリストレージにあり、Googleへの外部リダイレクト後にStreamlitの新しい実行セッションから参照できませんでした。

### 対策

- verifierをサーバー側の期限付き一時DBへ保存
- URLにはverifierではなくランダムな `oauth_flow` だけを載せる
- verifierがない場合は交換しない
- 成功・失敗後に一時データとクエリパラメータを削除
- 古いコールバックURLを再利用せずログインからやり直す

関連: `local_web_app/services/auth_service.py`

## Google Calendar APIが403 Forbidden

### 症状

Googleログインは成功するが、Calendar取得で403になります。

### 確認済みコード

- OAuth scopeに `https://www.googleapis.com/auth/calendar.readonly` を要求
- Calendar APIにはSupabaseセッションの `provider_token` をBearer tokenとして送信

### 主な原因候補

- Google CloudでCalendar APIが有効になっていない
- OAuth同意画面のData AccessにCalendar scopeが反映されていない
- 古い同意で発行されたtokenを使用している
- 対象CalendarへのGoogleアカウント権限がない
- Google provider tokenが期限切れ

### 対策

1. Google Cloudで対象Projectを確認する
2. Calendar APIがEnabledか確認する
3. Data Accessのscopeを確認する
4. アプリからログアウトする
5. Googleアカウントの接続済みアプリから必要に応じて同意を解除する
6. 再ログインし、Calendar読取権限へ同意する

トークンを画面やログへ出力しないでください。

## duplicate key: payments_pkey

### 症状

`受領・押印済み` で次が発生します。

```text
duplicate key value violates unique constraint "payments_pkey"
```

### 原因

SQLiteから `payment_id` を明示して移行した一方、PostgreSQL Identityシーケンスが最大IDへ進んでいませんでした。

### 対策

既存ProjectではSQL Editorで次のファイルを実行します。

```text
local_web_app/supabase_sequence_sync.sql
```

以後、`migrate_to_supabase.py` は移行成功後に `sync_migration_identity_sequences()` を自動実行します。

## 他テーブルのduplicate key

### 原因

`students`、`charges`、`audit_logs` も明示ID付き移行のため、同じ問題が起こり得ます。

### 対策

`sync_migration_identity_sequences()` は次をまとめて同期します。

- `students.student_id`
- `charges.charge_id`
- `payments.payment_id`
- `audit_logs.log_id`

## StreamlitがWelcomeメール入力で停止

### 症状

Community Cloudでポート8501が起動せず、ヘルスチェックに失敗します。

### 原因

起動ファイルが `local_web_app/app.py` のため、ルート側設定だけではStreamlitが設定を読まない場合がありました。また、ローカル設定に `headless=false` とloopback addressが残っていました。

### 対策

`local_web_app/.streamlit/config.toml` を確認します。

```toml
[browser]
gatherUsageStats = false

[server]
headless = true
```

`address = "127.0.0.1"` をCloud設定へ入れません。

## PowerShell環境変数が見えない

### 症状

設定したはずの `SUPABASE_URL` や `SUPABASE_SERVICE_ROLE_KEY` が未設定と表示されます。

### 原因

PowerShellの `$env:...` は設定したプロセスと、その子プロセスにだけ引き継がれます。別ターミナルや別セッションでは見えません。

### 対策

移行コマンドと同じPowerShellで設定します。値は表示しません。

```powershell
$env:SUPABASE_URL="取得した値"
$env:SUPABASE_SERVICE_ROLE_KEY="取得した値"
python migrate_to_supabase.py --commit
```

終了後はPowerShellを閉じます。

## 仮想環境のPythonが見つからない

### 症状

`.venv` が以前のPythonインストール先を参照し、起動できません。

### 対策

Pythonを確認し、必要なら仮想環境を作り直します。

```powershell
python --version
Remove-Item -Recurse .venv
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
```

削除前に対象パスが `local_web_app/.venv` であることを必ず確認します。

## Googleログイン後に許可されない

### 原因候補

- Streamlit Secretsの `app.allowed_emails` にない
- Supabase `allowed_teachers` にない
- 大文字小文字や別Googleアカウントを使用している

### 対策

両方へ同じGoogleアカウントを登録します。実メールをdocsやGitへ書きません。

## 切り分けの基本

1. `git status` でコード状態を確認
2. Community CloudのDeploy logsを確認
3. Secret値そのものではなく「設定済みか」だけ確認
4. Supabaseは最初にSELECTで状態確認
5. 問題を再現する最小操作を記録
6. 修正後にpytestと本番スモークテストを実施
